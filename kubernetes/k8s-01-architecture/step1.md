# Шаг 1: Общая схема кластера

Любой кластер K8s делится на две части: **Control Plane** (мозг) и **Worker Nodes** (мускулы).

```text
+---------------------------------------------------+
|                 ПОЛЬЗОВАТЕЛЬ (kubectl)            |
+-------------------------+-------------------------+
                          | (REST API)
                          v
+---------------------------------------------------+
|                CONTROL PLANE (Master)             |
|                                                   |
|  [ API Server ] <-------+------> [ etcd ]         |
|        ^                |        (База данных)    |
|        |                v                         |
|        +---- [ Scheduler ]                        |
|        |       (Выбирает узлы для контейнеров)    |
|        |                                          |
|        +---- [ Controller Manager ]               |
|                (Поддерживает нужное состояние)    |
+-------------------------+-------------------------+
                          |
                          v
+---------------------------------------------------+
|                   WORKER NODE                     |
|                                                   |
|  [ Kubelet ] <---> [ Container Runtime ]          |
|  (Агент K8s)         (например, containerd)       |
|                              |                    |
|                              v                    |
|                       [ POD 1 ] [ POD 2 ]         |
|                                                   |
|  [ Kube-proxy ] (Сетевые правила / iptables)      |
+---------------------------------------------------+
```

## Базовые понятия

- **Node (Узел)** — физический или виртуальный сервер.
- **Pod (Под)** — минимальная единица в K8s. В 90% случаев в одном поде крутится ровно 1 контейнер (например, nginx). K8s не управляет контейнерами напрямую, он управляет Подами.
